# OmniMemory

> 基于 markdown 文件树的跨会话持久记忆系统。任何 AI agent 阅读本文档即可接入。

---

## Agent 使用协议

### Session 开始

1. **阅读本文档**，获取用户全局信息、规则和文档索引
2. **匹配当前工作区**：用当前工作目录（cwd）与下方「工作区索引」中的路径比对
   - 匹配到 → 阅读该工作区的 `README.md`（规则）和 `progress.md`（进展与 TODO）
   - 未匹配到 → 询问用户这是哪个项目，确认后创建新工作区（见「新文档创建规则」）
   - 路径比对时注意归一化：`E:/` = `/e/` = `E:\`，忽略大小写和末尾斜杠
3. **简要向用户报告**你已了解的上下文（项目状态、待办、需要注意的事项）

### Session 进行中

- **执行任何 skill 或生成格式化内容前**，先读 `global/preferences.md` 中的偏好设置，直接按偏好格式生成，避免二次修改浪费 token
- **建议方案、库、工具前**，先查 `global/graveyard.md` 是否有相关失败记录。如有，警告用户再做决定
- **遇到需要特定指南的任务**（如发布 release、部署等），先在当前工作区目录和 `global/skills/` 中查找对应文档。找到则遵循，未找到则完成任务后创建

### Session 结束 / 任务完成后

每次回答完用户的问题或完成一项工作后，执行以下检查：

1. **判断是否有需要更新的内容**：
   - 新的决策、规则、约定 → 更新工作区 README.md 或创建子文档
   - 工作进展变化 → 更新工作区 progress.md
   - 发现新的失败经验 → 更新 global/graveyard.md
   - 用户偏好变化 → 更新 global/preferences.md
   - 学到新的可复用技能 → 创建 global/skills/ 或工作区 skills/ 子文档
   - 新建了工作区或技能文档 → 更新本文档（OMNI.md）的对应索引表
2. **如果有更新**：修改对应文档，然后自动 git commit（见下方规则），告知用户：`已更新 xx、xx、xx`
3. **如果没有更新**：告知用户：`当前 OmniMemory 没有需要更新的地方`

### 新文档创建规则

- **适用于所有工作区**的规则或技能 → 放入 `global/` 或 `global/skills/`，更新 OMNI.md 技能索引
- **仅适用于当前工作区**的规则 → 放入该工作区目录下，更新工作区 README.md 的子文档索引
- **规则内容过长**（超过 ~20 行）→ 拆分到子文档，原位替换为：`→ 详见 [文档名](./path.md)`
- **新建工作区**时，创建 `workspaces/{id}/README.md` 和 `workspaces/{id}/progress.md`，并更新 OMNI.md 工作区索引

**新工作区 README.md 模板**：
```
# {项目名}
## 项目概要
- 描述：
- 技术栈：
- 仓库地址：
## 规则与约定
（本工作区专属的规则）
## 子文档索引
| 文档 | 路径 | 简介 |
|---|---|---|
| 进展记录 | [progress.md](./progress.md) | 已完成进展和 TODO |
```

**新工作区 progress.md 模板**：
```
# {项目名} - 进展记录
## 已完成
（最新在前）
## TODO（按优先级排序）
（高优先级在前）
```

### 自动 Git Commit

每次更新 OmniMemory 中的任何文档后，在 OmniMemory 仓库根目录执行：
```bash
cd {OmniMemory 根目录}
git add -A
git commit -m "[OmniMemory] 更新 {变更文件列表}"
```

### 树形递归原则

每个文档既是内容载体，也可以是子文档的索引。当某条内容在当前文档中篇幅过长时：
1. 创建子文档，将详细内容移入
2. 原文档中替换为一行索引链接
3. 更新最近的索引表
